# 04 — QA

### Checklist de release (antes de cada merge a `main` que toque UI o functions)

- [ ] CI verde (unitarias + contratos + integración).
- [ ] Probado en móvil (viewport 390px) y desktop.
- [ ] Modo claro/oscuro del sistema no rompe legibilidad (si aplica).
- [ ] Estados: cargando / fresco / stale / error visibles y correctos.
- [ ] Disclaimers presentes (footer + junto a predicción + en chat).
- [ ] Sin `console.log` de debug ni claves en el cliente.
- [ ] Números redondeados; horas en CDMX etiquetadas.
- [ ] Lighthouse ≥85 performance, ≥90 accesibilidad.
- [ ] `docs/` actualizado si cambió arquitectura o contratos (Doc-Scribe).

### Monitoreo post-deploy (QA continuo)

| Qué | Cómo | Frecuencia |
|---|---|---|
| Cron horario vivo | Ver `generated_at` de `latest.json` (alerta si >2h viejo) | Diario, vistazo |
| Entrenamiento diario | Badge de estado del workflow en el README | Diario |
| Salud de datos | `metrics/health.json` renderizado en `/status.html` interna | Semanal |
| Consumo Groq | Contador global diario en Blobs + headers `x-ratelimit-remaining-*` logueados | Semanal |
| Accuracy real vs. mostrada | Verificación cruzada manual del rolling | Al cierre de cada fase |

### Gestión de defectos

- Bugs como Issues de GitHub con etiquetas `bug`, `fase-N`, severidad (`bloqueante/mayor/menor`).
- **Bloqueante** = rompe el sitio, corrompe datos o expone secretos → hotfix inmediato.
- **Mayor** = funcionalidad degradada con workaround → próxima sesión de trabajo.
- **Menor** = cosmético → backlog.
